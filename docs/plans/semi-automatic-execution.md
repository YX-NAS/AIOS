# AIOS 半自动执行方案

## 目标

先把 AIOS 固化成“任务中枢 + 执行指引生成器”：

1. 目标输入后自动拆分任务；
2. 每个子任务给出推荐模型和验收标准；
3. 生成可直接复制给模型的 Context Pack；
4. 人工手动切换 `ccswitch` 到对应模型；
5. 在 Codex / Claude Code 中完成开发；
6. 完成后回写任务状态、记录和总结。

## 当前边界

- 不自动调用 `cc-switch`。
- 不自动调用 Codex / Claude Code。
- 不自动修改业务代码。
- 不自动提交 Git。

## 当前可落地流程

1. `aios init` 初始化项目。
2. `aios scan` 扫描项目并刷新索引。
3. `aios task plan "目标"` 拆解任务。
4. 选择任务后用 `aios route TASK-ID` 查看推荐模型。
5. 用 `aios pack TASK-ID --model <模型>` 生成 Context Pack。
6. 手动在 `ccswitch` 中切换到对应模型。
7. 把 Context Pack 交给 Codex / Claude Code 执行。
8. 本地跑测试并修复。
9. 用 `aios complete TASK-ID --summary "..."` 回写结果。

## 为什么先做半自动

这个阶段最稳妥。

- `cc-switch` 官方产品目前仍是桌面端优先，自动化接口不是当前最稳的依赖点。
- `aios` 已经具备任务拆分、路由建议、Context Pack 生成和完成回写能力。
- 先把人肉切换跑通，可以尽快验证“任务管理是否真的有用”。

## 后续自动化方向

等半自动流程稳定后，再做自动调度层：

1. `ccswitch` 能力适配器；
2. 执行器适配器；
3. 任务完成回收器；
4. 模型切换日志；
5. 自动失败回退；
6. 测试通过后再允许自动完成回写。

## 风险

- 模型切换和 CLI 工具状态可能受平台影响。
- 自动化一旦接入，最容易出问题的是“切换成功但工具没刷新配置”。
- 所以第一阶段必须保留人工确认。

## 验收标准

- 用户能从一个目标开始，拿到拆分后的任务列表。
- 每个任务都能生成推荐模型和 Context Pack。
- 人工切换 `ccswitch` 后可以完成一次完整开发闭环。
- 完成结果能回写到 AIOS。
