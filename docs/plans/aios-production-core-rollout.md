# AIOS 第一生产力与系统管理核心落地规划

生成时间：2026-07-06

## 1. 定位结论

AIOS 不应再定位为“多模型开发辅助工具”，而应定位为个人与团队级的 **AI 生产操作台**。

它在你的实际工作体系中的角色应是：

- 统一登记所有项目、服务、模型、执行器和关键入口；
- 统一承接需求、问题、待办、自动化任务和交付记录；
- 统一生成任务上下文、模型路由、执行记录和验收摘要；
- 统一管理多项目、多模型、多工具之间的切换和续接；
- 逐步从半自动管理台升级为受控自动执行中枢。

一句话：

> AIOS 是你的第一工作入口，Codex、Chatbox、Ollama、ComfyUI、ccswitch、GitHub、服务器只是被 AIOS 调度和记录的执行资源。

## 2. 当前基础判断

当前 AIOS 已具备成为生产核心的底座：

- CLI 与 Web UI 已存在；
- 多项目 launcher 已存在；
- `.aios/` 项目知识目录已存在；
- 任务、模型路由、Context Pack、handoff、执行记录已存在；
- 自动执行、自动完成、自动 commit / push / PR、接管队列已有原型；
- 文档、测试、操作手册已有基础。

当前主要缺口不是“再做一个大功能”，而是：

1. 没有被强制设为日常第一入口；
2. 真实项目接入不够系统；
3. 项目状态、系统资源、模型服务还没有统一看板；
4. 自动化能力没有绑定到你的真实工作节奏；
5. 完成标准还没有成为所有项目的统一制度。

## 3. 落地原则

### 3.1 先做生产制度，再做高级自动化

AIOS 要先管住这些稳定对象：

- 项目
- 目标
- 任务
- 上下文
- 执行器
- 模型
- 验收
- 交付记录
- 系统资源

如果这些对象不稳定，继续堆自动执行只会放大混乱。

### 3.2 先接入真实项目，不再只用测试项目验证

AIOS 的第一批生产项目应直接使用你的真实主线：

1. `xiazhi-ai.git`
2. `AIOS`
3. `codex-deepseek-lifeline`
4. `ecosystem-hub`
5. `API_Router`
6. `wxautomation` 或 `WX-autosys`

每个项目必须有 `.aios/`，并形成最小项目档案。

### 3.3 半自动是当前主路径，全自动只跑低风险任务

当前生产策略：

- 高风险任务：AIOS 生成 Pack + handoff，人工确认后由 Codex 执行；
- 中风险任务：AIOS 调起 CLI 执行器，完成后人工验收；
- 低风险任务：允许自动执行、自动测试、自动回写；
- 涉及部署、付费、密钥、生产数据的任务必须有人确认。

### 3.4 所有完成必须可追溯

一个任务只有满足以下条件才算完成：

- 有任务记录；
- 有执行记录；
- 有实际模型或执行器记录；
- 有测试或验证结果；
- 有完成摘要；
- 必要时有 Git commit / PR / Release。

## 5. 并入实际生产的主流程

### 5.1 每天打开的第一个系统

日常工作从 AIOS Launcher 开始：

```bash
cd /Users/yaxun/SynologyDrive/日常工作/Github/AIOS
source .venv/bin/activate
aios launcher --port 8755
```

固定访问：

```text
http://127.0.0.1:8755
```

AIOS Launcher 首页必须回答 5 个问题：

1. 今天最重要的任务是什么；
2. 哪些项目正在进行；
3. 哪些任务被阻塞；
4. 哪些执行需要验收；
5. 哪些系统或模型服务不可用。

### 5.2 任何新工作先进入 AIOS

以后不要直接在 Codex 里开始一个复杂任务。标准入口：

1. 在 AIOS 中选择项目；
2. 创建目标或任务；
3. 让 AIOS 拆解任务；
4. 生成 Context Pack；
5. 选择推荐模型；
6. 执行；
7. 回写结果。

极小任务可以跳过，但超过 30 分钟、跨天、需要部署、需要多模型接力的任务必须进入 AIOS。

### 5.3 Codex 变成执行器，不再是任务记忆主库

Codex 的角色：

- 实施代码修改；
- 调试；
- 测试；
- 生成 PR / Release 内容；
- 处理单次任务执行。

AIOS 的角色：

- 记录为什么做；
- 记录做到哪一步；
- 记录用哪个模型做；
- 记录验证结果；
- 记录下一步。

这样可以避免项目状态只存在于 Codex 对话历史里。

## 7. 版本落地路线

### 阶段 1：生产接入版

目标：AIOS 成为每天打开的第一入口。

范围：

- 注册第一批真实项目；
- 每个项目初始化 `.aios/`；
- 每个项目生成项目档案；
- launcher 首页展示项目状态；
- 建立“今日任务 / 本周重点 / 阻塞任务”视图；
- 建立任务完成统一标准。

验收：

- 至少 5 个真实项目接入；
- 每个项目能看到任务、最近执行、最近验证；
- 能从 launcher 找到当天最重要任务。

### 阶段 2：生产执行版

目标：AIOS 不只管任务，还能稳定调起执行。

范围：

- 固化 `codex-cli` 执行器；
- 支持手动 / CLI 两种执行路径；
- 执行结果自动写回；
- 失败进入人工接管队列；
- 支持项目级验证命令；
- 支持低风险任务自动完成。

验收：

- 能在真实项目中完成一次“任务创建 → 执行 → 测试 → 回写”；
- 失败任务不会静默丢失；
- 人工接管队列可见。

### 阶段 3：系统管理版

目标：AIOS 接管系统资源管理。

范围：

- 记录 Ubuntu AI 主机状态；
- 记录 Ollama / ComfyUI / Chatbox 状态；
- 记录 Tailscale / LAN 可达性；
- 记录模型清单和能力；
- 与 ecosystem-hub 的资源台账打通；
- 在 AIOS 中显示“系统是否可用于生产”。

验收：

- 能看到本地 AI 服务是否可用；
- 能看到推荐使用哪个模型；
- 能看到 ComfyUI / Ollama 入口；
- AIOS 能提示资源不可用原因。

### 阶段 4：受控自动化版

目标：让 AIOS 处理稳定、重复、低风险的执行链路。

范围：

- 自动选择 ready 任务；
- 自动选择模型；
- 自动生成 Pack；
- 自动调用执行器；
- 自动运行测试；
- 测试通过后自动回写；
- 可选自动 commit / push / PR；
- 失败自动降级或进入接管。

验收：

- 至少一个真实项目可跑完整低风险自动 pipeline；
- 自动执行有预算和重试上限；
- 自动执行不会影响生产服务。

## 8. 日常工作制度

### 8.1 每日开工 5 分钟

在 AIOS Launcher 中完成：

- 查看今日任务；
- 查看阻塞任务；
- 查看系统资源状态；
- 选择当天最多 3 个重点；
- 明确第一件要执行的任务。

### 8.2 每个任务开始前

必须确认：

- 项目是否选对；
- 任务目标是否清楚；
- 验收标准是否存在；
- 推荐模型是否合理；
- 是否需要生产保护。

### 8.3 每个任务结束后

必须回写：

- 完成摘要；
- 实际模型；
- 测试命令；
- 测试结果；
- 是否部署；
- 下一步。

### 8.4 每周复盘

AIOS 每周应能输出：

- 完成任务；
- 未完成任务；
- 卡点；
- 模型成本；
- 失败原因；
- 下周重点。

## 10. 风险与控制

| 风险 | 表现 | 控制方式 |
|---|---|---|
| 工具过重 | 小任务也被迫走流程 | 30 分钟以下任务允许跳过 |
| 自动化误操作 | 未确认就改生产 | 生产任务默认人工确认 |
| 上下文污染 | 多项目任务混在一起 | 一项目一 `.aios/`，launcher 只做索引 |
| 模型切换不稳定 | Codex 历史、插件、配置异常 | AIOS 保存 Pack 与执行记录，不依赖单一聊天历史 |
| 维护成本高 | 功能太多但不用 | 优先真实项目接入和每日入口 |
| 资源状态不准 | Tailscale / LAN 状态漂移 | 服务探测必须区分 LAN 与 Tailscale |

## 12. 成功标准

当以下条件满足时，可以认为 AIOS 已经进入你的实际生产体系：

- 每天工作从 AIOS Launcher 开始；
- 主要项目都在 AIOS 中有状态；
- 复杂任务不再只存在于聊天记录；
- 每个任务能看到开始、执行、验证、完成；
- 切换模型不再导致项目上下文丢失；
- Ubuntu AI、Codex、Chatbox、ComfyUI 等资源能被统一查看；
- 每周能从 AIOS 直接生成工作复盘；
- 至少一个真实项目能跑通低风险自动执行闭环。
