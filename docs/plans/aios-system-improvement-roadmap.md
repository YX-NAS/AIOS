# AIOS 系统补强与痛点解决规划

生成时间：2026-07-02

## 目标

把“同类项目可借鉴能力”收敛成 AIOS 自己的后续开发路线，优先解决当前最影响可用性和可持续扩展的痛点。

这份规划只回答 3 个问题：

1. AIOS 现在最痛的点是什么；
2. 对标项目里哪些能力值得借；
3. 下一阶段应该按什么顺序补，做到什么算完成。

## 非目标

这轮规划不做：

- 直接上桌面 GUI 自动点按方案；
- 直接引入复杂多代理自治运行时；
- 先做分布式、多机 mesh、云原生调度平台；
- 为“未来可能会用到”提前堆大框架。

## 当前最核心的 5 个痛点

### 1. 执行层断在系统外

当前 AIOS 已经能管：

- 任务
- 路由
- Pack
- handoff
- execution record

但还不能真正接管：

- 调起哪个执行器
- 用哪个 CLI 执行
- 执行结果如何采集
- 失败后如何重试或回退

这导致 AIOS 现在更像“执行前后管理器”，还不是“执行中枢”。

### 2. 任务拆解深度不够

当前任务拆解已经可用，但还偏：

- 平面任务
- 模板拆解
- 没有依赖关系
- 没有草案确认层

一旦目标复杂，用户就很难靠当前任务系统表达“先做什么、谁依赖谁、做到哪一步算通过”。

### 3. Context Pack 够用但不够强

当前 Pack 已经能支撑半自动流程，但还缺：

- 分层上下文
- 更强相关文件筛选
- Pack 质量检查
- 更清晰的 token / 上下文窗口约束

这会直接影响模型执行质量。

### 4. 执行状态可见，但执行全貌不可见

当前有任务状态和 execution record，但 launcher 还看不到完整执行画像，例如：

- 当前项目绑定的执行器
- 当前推荐模型与实际模型偏差
- 最近失败原因
- 最近测试结果
- 最近交付时间

多项目一多，还是容易重新乱掉。

### 5. 主路径仍然偏散

虽然已经有 `run --manual` 作为主入口，但新用户仍容易困惑这些入口的关系：

- `task plan`
- `pack`
- `handoff`
- `run --manual`
- `ccswitch export`
- `complete`

系统已经能做很多事，但“推荐操作路径”还不够一眼清楚。

## 借鉴原则

基于对比文档，后续借鉴遵循 4 条原则：

1. **借能力，不抄架构**  
参考能力边界和落地方式，不照搬别人的大平台结构。

2. **先补执行，再补花活**  
优先解决执行适配、任务树、上下文质量，不先堆代理讨论系统。

3. **CLI 优先，不走 GUI 自动化优先**  
只要有稳定 CLI，就不把桌面 GUI 自动化当主路线。

4. **每一步都要能回退到当前半自动方案**  
任何自动化改造，都不能破坏现有 `run --manual` 流程。

## 借鉴能力映射

| 借鉴方向 | 对标项目 | AIOS 应吸收什么 |
|---|---|---|
| 执行适配器 | `claude-mpm`、`AI-Agents-Orchestrator` | 统一 executor adapter、CLI-first、session lifecycle 设计 |
| 任务树 | `plan-cascade` | 父子任务、依赖关系、拆解草案确认 |
| Context 工程 | `agent-context` | 分层 Pack、bounded search、Pack 校验 |
| 可观测能力 | `ai-maestro`、`AI-Agents-Orchestrator` | 实例总览、执行状态、失败摘要、最近测试和交付信息 |

## 路线图总览

后续规划按 4 个阶段推进。

### 阶段 A：执行适配基础层

目标：让 AIOS 真正开始接近“执行中枢”。

### 阶段 B：任务树与拆解增强

目标：让 AIOS 能表达复杂任务，而不只是记录平面任务列表。

### 阶段 C：Context Engine 补强

目标：让生成给模型的上下文更稳、更相关、更可验证。

### 阶段 D：执行总览与交付闭环

目标：让多项目、多任务、多模型协作真正可观测、可复盘。

## 阶段 A：执行适配基础层

### 目标

建立统一执行器抽象，让 AIOS 不再只停留在“导出材料”，而能受控地进入“调起执行器”的阶段。

### 范围

先只做 CLI-first 原型，不碰桌面 GUI 自动点按。

### 开发任务

1. 定义 executor adapter 接口  
完成标准：统一抽象 `prepare / start / status / finish / cancel` 等最小生命周期。

2. 增加执行器配置层  
完成标准：全局或项目配置中可声明可用执行器，如 `codex-cli`、`claude-code-cli`、`manual`。

3. 增加手动执行与 CLI 执行两种模式  
完成标准：保留现有 `run --manual`，新增受控 `run --executor <id>` 原型入口。

4. 增加执行器回写协议  
完成标准：执行器能统一回写 stdout/stderr 摘要、退出状态、开始/结束时间。

5. 失败回退到手动执行  
完成标准：CLI 执行异常时，任务仍可回退到半自动人工执行，不破坏 execution record。

### 验收标准

- 不破坏现有半自动流程；
- 至少接通 1 个稳定 CLI 执行器原型；
- execution record 能区分 `manual` 与 `cli`；
- 失败时有明确错误而不是静默中断。

## 阶段 B：任务树与拆解增强

### 目标

让 AIOS 具备更强的需求拆解能力，减少“只有目标确认、没有真正任务结构”的问题。

### 开发任务

1. 支持父任务 / 子任务结构  
完成标准：任务数据结构支持层级关系。

2. 支持任务依赖关系  
完成标准：能表达“必须先完成 A 再做 B”。

3. 引入拆解草案层  
完成标准：`task plan` 先产出草案，确认后再正式写入任务。

4. Web UI 支持拆解结果编辑  
完成标准：用户可在落盘前修改标题、验收标准、优先级和模型建议。

5. 增加阶段性验收点  
完成标准：一个大目标下可以分阶段标记完成。

### 验收标准

- 复杂目标不再只生成 3 条平面任务；
- 拆解结果在写入前可调整；
- 子任务与依赖关系在 Web UI 可见。

## 阶段 C：Context Engine 补强

### 目标

让 Pack 和 handoff 从“可用”变成“更稳、更相关、更少跑偏”。

### 开发任务

1. 引入分层 Pack  
完成标准：至少区分项目层、任务层、相关文件层。

2. 增强相关文件筛选  
完成标准：结合任务类型、标题关键词、近期变更文件、手动勾选结果生成上下文。

3. 增加 Pack 校验  
完成标准：缺验收标准、缺相关文件、上下文过长时给出警告。

4. 增加 token / 窗口预估  
完成标准：生成 Pack 时输出估算与超限提示。

5. 明确 Pack 与 handoff 分工  
完成标准：文档和 UI 中明确 Pack 负责上下文，handoff 负责执行说明。

### 验收标准

- Pack 结构更稳定，用户能看出每层信息从哪来；
- 大仓库下无关文件噪声明显减少；
- 超长上下文能提前提示，而不是执行时才发现。

## 阶段 D：执行总览与交付闭环

### 目标

让 launcher 从“项目入口页”升级为“项目执行总览页”。

### 开发任务

1. 增强项目摘要  
完成标准：显示当前执行器、最近执行模型、最近测试结果、最近交付时间。

2. 增加失败摘要  
完成标准：项目卡片可见最近一次失败原因或失败状态。

3. 增加测试与交付面板  
完成标准：单项目页可看最近测试命令、测试结果、完成总结。

4. 增加执行历史列表  
完成标准：每个项目可查看最近若干次执行记录。

5. 增加主路径引导  
完成标准：首页和单项目页都能更清楚地告诉用户“下一步该做什么”。

### 验收标准

- 多项目并行时能快速看出哪个项目卡在什么地方；
- 用户不需要翻文件就能看到最近一次执行和测试结论；
- 新用户能沿主路径完成一次完整任务闭环。

## 实施优先级

按优先级排序如下：

1. 阶段 A：执行适配基础层
2. 阶段 B：任务树与拆解增强
3. 阶段 C：Context Engine 补强
4. 阶段 D：执行总览与交付闭环

原因：

- 不先补执行层，AIOS 很难跨过“只是管理壳”的阶段；
- 不补任务树，复杂目标拆解会持续成为硬伤；
- 不补 Context，执行质量会上不去；
- 可观测和总览是放大器，应该建立在前面几层已经稳定的基础上。

## 本阶段建议立即启动的具体工作

建议把下一轮开发正式收口为以下 3 项：

1. `P3-3A`：executor adapter 原型  
先接 1 个 CLI 执行器，证明 AIOS 能真正调起执行。

2. `P3-4A`：task plan 草案确认层  
解决“拆解太浅、落盘太快”的直接痛点。

3. `P3-5A`：Pack 分层 + token 预估  
先把上下文质量做稳，再扩大自动执行范围。

## 测试规划

### 自动化测试

1. executor adapter 生命周期测试  
覆盖 prepare / start / status / finish / error fallback。

2. 任务树数据结构测试  
覆盖父子关系、依赖关系、草案确认与取消。

3. Pack 校验测试  
覆盖空验收标准、缺关键文件、超长上下文警告。

4. launcher 摘要测试  
覆盖最近执行状态、测试结果、失败摘要显示。

### 手工验收

1. 创建一个复杂目标；
2. 先生成拆解草案并修改后确认；
3. 启动一次执行器原型或回退到手动执行；
4. 生成并检查分层 Pack；
5. 完成执行并回写结果；
6. 在 launcher 中确认执行摘要、测试结果和项目状态已刷新。

## 决策结论

AIOS 下一阶段不应该继续优先投入：

- GUI 自动化切模型
- 更花哨的多代理对话界面
- 重型分布式基础设施

而应该围绕 3 条主线补强：

- **执行器适配**
- **任务树拆解**
- **Context Engine**

把这 3 条线补齐后，AIOS 才会真正从“半自动管理台”走向“可持续扩展的开发操作系统”。
