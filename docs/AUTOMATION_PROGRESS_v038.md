## 二、当前全自动化瓶颈（6 个关键缺口）

### 缺口 1：ccswitch 外部状态不可读取 🔴

系统能导出 Deep Link 和 Session Handoff，能桥接到终端，但**无法获知用户是否真的完成了外部切换**。Bridge 确认目前依赖人工点击"确认桥接成功"按钮。

**影响**：这是自动化链条上唯一需要人工操作的环节，阻断了从"拆解任务"到"审查通过"的全自动链路。

**解决路径**：
- 短期：bridge signal 自动确认默认开启，配合更严格的预检（检测到恢复命令已执行 + 终端窗口存在 → 自动确认）
- 长期：如果 `ccswitch` 或 `codex` 提供 CLI 状态查询接口（`codex session list --json`），AIOS 可以直接探测

### 缺口 2：任务拆解结果不可自动调整 🟡

当前 `task plan` 能按模板生成拆解草案，也支持 draft 确认流程，但草稿质量完全依赖模板规则。如果拆分不合理（模块划分错误、依赖关系缺失），系统不会自动修正。

**影响**：拆解质量直接影响后续自动派发的执行质量，garbage in → garbage out。

### 缺口 3：自动交付结论不能自动生成 🟡

`--auto-finish` 仍然要求 `--summary "..."` 由人工提供。系统不会从 git diff + 测试结果 + 执行日志自动生成交付总结。

**影响**：每条任务完成都需要人工写一句话。对于批量小任务，这个开销累计很大。

### 缺口 4：多步骤工作流没有编排引擎 🔴

`aios run auto` 只派发一条 ready 任务，真实开发中的"编码→测试→审查→提交"多步流水线需要人工逐步触发。

**影响**：没有工作流编排，AIOS 的自动化只能单步进行，做不到"一句话启动一个完整开发流程"。

### 缺口 5：审查结果不会自动触发修复 🟡

`aios review create` 创建的审查任务需要人工阅读审查结论，然后手动触发修复重试。审查不通过 → 自动重新派发修复的链路还没有打通。

### 缺口 6：没有测试 🟡

`python -m pytest` 返回 `No module named pytest`。整个项目没有任何自动化测试。P0-1 在迭代计划中列了但一直被延后。
