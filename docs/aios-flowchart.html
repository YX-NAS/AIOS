AIOS 使用流程

从零到全自动交付的完整路径

一、项目初始化 & 准备

flowchart TD
    A["📁 进入项目目录"] --> B["aios init --name my-project"]
    B --> C["aios scan"]
    C --> D["aios repo map"]
    D --> E["aios repo index"]
    E --> F["✅ 项目已就绪
.aios/ 目录创建完成
文件索引 + 仓库地图已生成"]

    style A fill:#f0f4ff,stroke:#4f46e5
    style F fill:#ecfdf5,stroke:#059669

做了什么：创建知识目录、扫描项目文件结构、生成仓库结构化地图和搜索索引，为后续智能文件选取打基础。

    G --> N["简单任务直接创建"]

做了什么：把自然语言需求拆成结构化任务树。复杂目标自动拆为"系统边界分析 → 接口设计 → 各模块实现 → 测试 → 文档"的任务链，每个子任务有依赖关系。

三、上下文准备（AIOS 独特价值）

flowchart TD
    P["📋 选择一个 ready 任务"] --> Q["aios route TASK-ID"]
    Q --> R["查看推荐模型和 fallback 链"]
    R --> S["aios pack TASK-ID --smart"]
    S --> T["生成 Context Pack"]
    T --> U{质量检查}
    U -->|相关度低| V["手动标注关键文件
优化任务标题关键词"]
    V --> S
    U -->|Token 超限| W["aios pack TASK-ID --budget 50000"]
    W --> S
    U -->|通过| X["✅ Context Pack 就绪
智能选取最相关文件
Token 在预算范围内
质量评分通过"]

    style P fill:#f0f4ff,stroke:#4f46e5
    style X fill:#ecfdf5,stroke:#059669

做了什么：不像纯 CLI 那样把整个仓库塞给模型。AIOS 根据任务类型和关键词智能选取最相关的文件，控制在目标模型的 token 预算内，生成质量评分。大仓库下这个步骤的收益最明显。

四、全自动执行（核心链路）

flowchart TD
    Y["🚀 启动自动执行"] --> Z["aios run auto
--executor codex-cli
--auto-finish
--verify-command 'pytest -q'
--auto-commit
--auto-push
--auto-pr"]
    Z --> AA["调度器选择下一条 ready 任务"]
    AA --> AB["AIOS 准备 Context Pack
拼接 handoff prompt"]
    AB --> AC["subprocess 调起 Codex CLI"]
    AC --> AD["Codex CLI 调用模型 API
模型执行编码/修复"]
    AD --> AE{执行结果}
    AE -->|成功| AF["AIOS 运行验证命令
pytest -q"]
    AE -->|失败| AG["AIOS 分析失败类型"]
    AG --> AH{失败分类}
    AH -->|验证失败| AI["切换到 fallback 模型
自动重新派发一次"]
    AH -->|网络/超时| AJ["同模型自动重试
受分类级次数上限控制"]
    AI --> AB
    AJ --> AB
    AF --> AK{验证结果}
    AK -->|通过| AL["aios run finish
自动填写完成记录"]
    AK -->|失败| AI
    AL --> AM["git commit
自动本地提交"]
    AM --> AN["git push
自动推送特性分支"]
    AN --> AO["GitHub Draft PR
自动创建"]
    AO --> AP["✅ 交付完成
执行记录 + changelog
+ memory 全部更新"]

    style Y fill:#f0f4ff,stroke:#4f46e5
    style AP fill:#ecfdf5,stroke:#059669

关键区别：这里 AIOS 不调模型 API。它通过 subprocess 调起 Codex CLI / Claude Code CLI，由那些 CLI 自己去调 API。AIOS 负责的是编排层——选任务、组上下文、调起执行器、监控状态、跑验证、处理失败、自动交付。

ccswitch 已被绕过：CLI 模式通过 --model 参数直接指定模型，不再需要人工在 ccswitch 中切换模型。调度器会自动推进，不卡在 bridge 确认步骤。

五、审查 & 闭环

flowchart TD
    AQ["👀 任务完成"] --> AR["aios review create TASK-ID
--focus correctness code_quality"]
    AR --> AS["自动创建审查任务
路由到不同模型"]
    AS --> AT["审查模型执行审查"]
    AT --> AU{审查结论}
    AU -->|approved| AV["✅ 任务最终交付"]
    AU -->|changes_requested| AW["任务状态回退 todo
记录审查反馈"]
    AW --> AX["aios run auto 重新派发
自动延续修复"]
    AX --> AQ

    style AQ fill:#f0f4ff,stroke:#4f46e5
    style AV fill:#ecfdf5,stroke:#059669

做了什么：审查用和实现不同的模型，避免"自己审自己"的盲区。审查不通过时自动回退任务状态，下一次 run auto 会自动拾取并修复。
